Hermes agent lab

Independent · active lab · 2026 · Windows + local/frontier models
ACTIVE LAB

Why I built it

Tool-using agents become risky when prompts, credentials, memory, and system authority collapse into one ambient trust boundary.

What I built

A native-Windows multi-agent appliance with 12 isolated role profiles, per-profile models, tools, state, and authority, a durable team work queue, and a canonical Obsidian knowledge layer.

One real workflow

A disposable manager-control run created a revision-specific review, routed requested changes into one corrected revision, required a new approval, created one dependency-bound non-live verification, and reconciled again without duplicating work.

How it fails safely

Finite retries, idempotency keys, atomic state, encrypted backups, hash-bound artifacts, explicit rollback, and supply-chain review for installed agent skills.

Engineering approach

Three rules that survive contact with production.

Deterministic core, model edge

Models interpret, search, rank, and draft. Code owns calculations, permissions, invariants, and mutations.

Evidence before authority

A successful-looking answer does not become system state until the relevant checks and authorization path agree.

Failure is a designed state

Timeouts, finite retries, idempotency, backup evidence, health checks, and rollback are part of the architecture.
